# InRecord 手機／平板 UI/UX 優化 — 設計規格

- **日期**：2026-06-15
- **方案**：A（沿用 CSS Modules）＋ 流體化（吸收 C 的零斷點跳動紅利）
- **範圍**：全站（手機 + 平板），分階段執行
- **視覺尺度**：修 RWD 為主，順手做適度視覺打磨；不做全站視覺重整
- **目標痛點**（使用者實測）：橫向溢出要左右捲、教室難用、互動元件難按、文字/按鈕太小

---

## 1. 設計原則

1. **逐頁可獨立驗收**：每個階段改完即可單獨在實機尺寸截圖驗收，不依賴後續階段。
2. **保留既有調校**：v3 兩層字體系統、CJK 斷行規則（`word-break:keep-all` + 禁則）、色票變數一律沿用。
3. **桌機零位移**：除非明確標示，桌機（≥1025px）版面與視覺維持現狀。
4. **手機/平板可與桌機分歧**：當分歧能換得更好的小螢幕體驗時，允許手機/平板採用不同編排（經使用者同意）。

## 2. 地基規範（Phase 0 落地，全站共用）

### 2.1 統一斷點

收斂目前 10+ 種雜亂斷點（480/600/640/700/760/768/900/960/980/1200…）為三段：

| 區間 | 寬度 | 用途 |
|------|------|------|
| 📱 手機 | `≤ 640px` | 單欄、全寬 sheet、漢堡選單 |
| 📲 平板 | `641 – 1024px` | 中間地帶，需專屬編排避免「桌機縮小版」 |
| 💻 桌機 | `≥ 1025px` | 維持現狀 |

- 慣例：手機用 `@media (max-width: 640px)`；平板用 `@media (min-width: 641px) and (max-width: 1024px)`；必要時 `@media (max-width: 1024px)` 表示「手機+平板」。
- 小手機細修（≤400px）保留為**例外**，僅在確有需要時加。
- CSS 無法在 `@media` 條件用變數，故斷點以「文件化慣例 + 逐檔收斂」落實，不是執行期變數。

### 2.2 流體 Token（新增於 `globals.css :root`）

```
--container-max: 1120px;
--container-pad: clamp(20px, 5vw, 44px);   /* 手機 20 → 桌機 44 */
--space-section: clamp(48px, 8vw, 80px);   /* 區塊上下留白，取代 80↔48 硬切 */
--tap-min: 44px;                            /* 最小觸控目標 */
```

字級流體化：標題沿用既有 `clamp()`；補上目前用固定值的內文/副標，改為 `clamp()` 平滑縮放。**不改既有已是 clamp() 的 Hero 標題級距。**

### 2.3 全域安全網（新增於 `globals.css`）

| 項目 | 作法 | 對應痛點 |
|------|------|----------|
| 防橫向溢出 | `html, body { overflow-x: clip; }` + 圖片/媒體 `max-width:100%`、長字 `overflow-wrap` 守門 | 左右捲動 |
| 最小觸控 44px | 按鈕/分頁/連結/圖示鈕點擊區 ≥ `--tap-min`；以 padding 或 min-height 達成 | 太小難點 |
| 瀏海/手勢安全區 | `env(safe-area-inset-*)`，固定底列/footer 加底部留白 | iPhone 底部吃字 |
| 觸控回饋 | 統一 `-webkit-tap-highlight-color: transparent` + `:active` 視覺回饋 | 觸感 |
| hover 防誤觸 | `@media (hover: none)` 停用位移型 hover（官網已有，推廣到全站元件） | 平板誤觸 |

### 2.4 統一容器類

新增 `.u-container { width: min(var(--container-max), calc(100% - var(--container-pad) * 2)); margin-inline:auto; }`，供各頁逐步替換現有零散的 `.container` 定義（官網 `.container` 行為對齊此規範）。

## 3. 分階段計畫

> 優先序＝痛點嚴重度 × 使用頻率。每階段獨立驗收。

### Phase 0 — 地基（前置，必先做）
- 在 `globals.css` 落地 §2.2 Token、§2.3 安全網、§2.4 容器類。
- **驗收**：全站快速掃一遍無回歸；360/390/768/1024px 無新增橫向溢出。

### Phase 1 — 教室 Classroom（最高優先）
**現況問題**：只有 900/600 兩斷點；手機把右側 280px 單元清單塞到最底、限高 300px，切換單元要捲過整個影片+分頁；分頁鈕字 13px、觸控區 ~12px。

- **單元清單行動化**：手機/平板把右側清單改為**可呼出的抽屜（drawer）或頂部摺疊**，讓學員不需捲到底就能切換單元。桌機維持右側欄。
- **播放器**：維持 16:9 `padding-top:56.25%`，確認手機全寬、無溢出。
- **影片資訊列**：標題/完成鈕在窄螢幕換行不擠；完成鈕觸控區 ≥44px。
- **分頁（留言/評分/作業）**：分頁鈕加大到 ≥44px 觸控、字級上調；分頁內容（留言、評分星、作業上傳 dropzone）手機間距與輸入體驗優化。
- **topbar**：≤600 已隱藏 email；確認登出鈕觸控區足夠。
- **驗收**：360/390/768px 可順暢切換單元、看影片、留言、上傳作業，全程無橫向捲動。

### Phase 2 — 互動練習元件（高優先）
**現況問題**：`PianoKeyboard` / `ChordKeyboard` 已用 `aspect-ratio 448/192` + `clamp()` 半流體化，但窄螢幕下鍵盤變得又矮又扁，黑鍵 ~28px 寬難精準點；`EarTraining` 同步檢查。

- **鍵盤觸控尺寸**：手機/平板設定**最小高度**或調整 `aspect-ratio`，確保白鍵/黑鍵在窄螢幕仍有足夠點擊高度；黑鍵點擊區加寬（視覺可不變、加大 hit area）。
- **多八度情境**：若鍵數多到單行過擠，評估手機改為**橫向捲動**或縮減顯示範圍（依元件實際資料決定）。
- **badge/標籤**：確認 `clamp()` 下標籤不重疊、可讀。
- **驗收**：360/390/768px 單指可準確點到目標鍵，連點無誤觸。

### Phase 3 — 官網 (/)（低風險細修 ＋ Hero 行動版重新設計）
**現況**：全站 RWD 最完整（已有漢堡選單、sticky 購買列、safe-area、平板帶、`hover:none`）。

低風險細修：
- 斷點 980/861 → 收斂至 1024，對齊全站。
- 補齊平板帶（768–1024）：curriculum、instructor、CTA 的留白與欄數，避免「桌機縮小版」。
- 防橫向溢出體檢（高風險點：方案緞帶 `right:-32px`、stats `translateY` 負位移）。
- 區塊 padding 由 640 硬切改 `--space-section` 流體過渡。
- 漢堡鈕觸控區 8px padding → ≥44px。

**Hero 行動版重新設計（重點，允許與桌機分歧）**：
- 重新編排「首頁最上方重疊的畫面」——目前手機下 Hero 內容（標題/影片卡/特色）堆疊後，與 stats 卡（負 `translateY` 上移）交界處易顯擠/重疊。
- 桌機 Hero **維持現狀**；手機/平板可採用不同的堆疊順序、間距、stats 卡銜接方式，做出乾淨、不重疊、層次清楚的首屏。
- **待確認**：spec 審查時與使用者確認「重疊」具體指 stats 卡上移銜接，或 Hero 內元素堆疊，或背景圖構圖——以鎖定改法。
- **驗收**：360/390/768/1024px 首屏無重疊/裁切，視覺乾淨；桌機零位移。

### Phase 4 — 購買／付款流程（中優先）
- `BuyModal` / `PreviewModal`：手機改為**全寬底部 sheet 或全螢幕**，按鈕 ≥44px，內容可捲不溢出。
- `/success`：手機留白與字級確認。
- **驗收**：手機完成「選方案→填資料→送出」無卡頓、無溢出。

### Phase 5 — 登入 ＋ 內容頁（中低優先）
- `/login`、`/classroom/login`：表單輸入體驗、Google 登入鈕觸控、`loginCard` 手機留白。
- `/contact`、`/privacy`、`/terms`：長文可讀性、`.content-card` 手機 padding（已有 640 處理，對齊新 Token）。
- **驗收**：手機表單好填、長文好讀。

### Phase 6 — 後台 Admin（最低優先）
- 2464 行管理介面以**平板可用**為目標（非手機完整最佳化）：表格橫向捲動容器、操作鈕觸控、分頁在平板可用。
- **驗收**：iPad 直/橫向可完成日常管理操作。

## 4. 驗收與測試策略

- **無自動視覺回歸基建**：以人工實機尺寸截圖為準。
- **標準檢查寬度**：360 / 390（手機）、768 / 1024（平板）、≥1280（桌機回歸）。
- **每階段檢查清單**：
  1. 無橫向捲動 / 無內容裁切。
  2. 主要互動觸控區 ≥44px。
  3. 桌機版面與視覺零位移（除明確分歧項）。
  4. CJK 斷行規則未被破壞。
- 既有單元測試（vitest）以邏輯為主，與本次純前端樣式改動無直接交集；不新增測試，改以截圖驗收。
